// 4.0 — The Detection Landscape //
Detection Methodology
Detection engineering is the systematic process of converting threat knowledge into reliable, low-false-positive detections. It sits at the intersection of threat intelligence, data engineering, and security operations.
David Bianco's Model — 2013
PYRAMID OF PAIN
How painful is it for the attacker if you detect and block this indicator? Hash blocks are trivial to defeat — TTPs require fundamental retooling.
TTPs Behavioral patterns — how adversaries operate. Hardest to change. Detection here forces fundamental retooling. TOUGH for attacker
TOOLS Specific malware families, C2 frameworks (Cobalt Strike, Sliver). Attacker must find/build alternatives. CHALLENGING
NETWORK IOCs C2 domains, IP addresses, URLs. Moderate cost to rotate infrastructure. ANNOYING
HASHES / IPs File hashes (MD5/SHA256), attacker IPs. Trivial to change — recompile or spin new VPS. TRIVIAL for attacker
Engineering principle: Invest heavily in TTP-level detections (behavioral). Hash and IP blocklists are necessary hygiene but insufficient defenses — attackers change them in minutes.
Detection Method 01
Signature-Based

Matches known-bad patterns: file hashes, byte sequences, domain names. Fast, cheap, high-fidelity for known threats.

Strengths:
  • Zero false positives on exact match
  • Extremely fast execution
  • Easy to share (IOC feeds)
Weaknesses:
  • Blind to new/modified variants
  • 1-bit change defeats hash match
  • Requires constant updates
Detection Method 02
Behavioral / Heuristic

Detects anomalous actions and sequences, not specific signatures. Catches zero-days and novel variants. The direction modern EDR is heading.

Strengths:
  • Catches unknown malware
  • Harder for attackers to evade
  • TTP-level detection
Weaknesses:
  • Higher false positive rate
  • Requires tuning per environment
  • More compute intensive
Detection Method 03
ML / Anomaly

Statistical deviation from baseline. Machine learning models trained on normal behavior flag outliers. Used in modern UEBA and next-gen EDR.

Strengths:
  • Detects subtle anomalies
  • No predefined signatures needed
  • Adapts to environment
Weaknesses:
  • Black box — hard to explain
  • Slow LttPD (slow attackers evade)
  • Requires large training baseline

Adversarial Awareness
EDR BYPASS TECHNIQUES — DEFENDER MUST KNOW
Understanding how attackers attempt to blind EDR allows defenders to build compensating controls and detection for the bypass attempts themselves.
BYPASS 1: Unhooking (direct syscalls) # EDR hooks ntdll.dll functions in user space # Attacker reads syscall numbers from disk copy of ntdll # Invokes syscall directly, bypassing EDR hooks # Detection: → Monitor ETW ETWTI provider (kernel-level) → ETWTI is not in ntdll — hooks don't help attacker → Alert on: unusual ntdll reads from disk → Alert on: module loaded from non-standard path → EDR kernel driver callbacks fire regardless of hooking
BYPASS 2: Process Injection into EDR # Inject code into the EDR agent process itself # EDR can't monitor its own memory writes # Mitigations: → Protected Process Light (PPL) on EDR processes → ObRegisterCallbacks: EDR denies any handle to itself → Tamper protection (CrowdStrike, Defender) → Memory integrity (HVCI/Virtualization-based security)
BYPASS 3: Kernel Driver Exploit # Use a vulnerable signed driver (BYOVD) # e.g. dell_hwinfo.sys, gdrv.sys # Get ring-0 execution → kill EDR driver # Detection: → Block vulnerable driver hashes (WDAC policy) → Alert on unusual driver loads (Sysmon Ev6) → MSFT vulnerable driver blocklist (updated) → Alert on EDR process termination events
BYPASS 4: Early Boot Execution # Execute before EDR driver loads # Bootkit in MBR/UEFI pre-loads malware # Mitigations: → Secure Boot (firmware signature chain) → UEFI attestation → Measured boot (TPM-based integrity) → ELAM driver (anti-malware early launch)

// 4.3 — SIEM Query Languages //
KQL & SPL Detection Queries
Production-ready queries for Microsoft Sentinel (KQL) and Splunk (SPL) covering the highest-value detection scenarios. Each query includes tuning notes and ATT&CK mapping.
KQL — Beaconing Detection (C2 Timing) T1071 · C2
// Detect regular beaconing patterns — statistical periodicity analysis // Tuning: adjust MinBeaconCount and StdDevThreshold for environment let LookbackWindow = 24h; let MinBeaconCount = 10; // min connections to flag let StdDevThreshold = 5.0; // low stddev = regular interval DeviceNetworkEvents | where TimeGenerated > ago(LookbackWindow) | where RemotePort in (80, 443, 8080, 8443) | summarize ConnectionCount = count(), IntervalSeconds = stdev(bin(TimeGenerated, 1s)) by DeviceName, InitiatingProcessFileName, RemoteIP, RemotePort | where ConnectionCount > MinBeaconCount | where IntervalSeconds < StdDevThreshold // low stddev = periodic | where InitiatingProcessFileName !in~ ( "chrome.exe", "firefox.exe", "msedge.exe", "svchost.exe" ) | project DeviceName, InitiatingProcessFileName, RemoteIP, RemotePort, ConnectionCount, IntervalSeconds | order by IntervalSeconds asc // most regular first
KQL — Lateral Movement via WMI/PsExec T1021 · Lateral Movement
// Detect lateral movement via WMI remote execution and PsExec patterns SecurityEvent | where EventID == 4688 // Process creation | where ParentProcessName contains "WmiPrvSE" | where NewProcessName !endswith in ( "mofcomp.exe", "wmiprvse.exe", "unsecapp.exe" ) | project TimeGenerated, Computer, Account, NewProcessName, CommandLine, ParentProcessName | union ( // PsExec pattern: PSEXESVC service creation SecurityEvent | where EventID == 7045 | where ServiceName contains "PSEXESVC" or ServiceFileName contains "\\ADMIN$\\" | project TimeGenerated, Computer, ServiceName, ServiceFileName ) | order by TimeGenerated desc
KQL — Impossible Travel Detection (UEBA) T1078 · Account Abuse
// Detect logons from geographically impossible locations within short time window SigninLogs | where ResultType == "0" // successful sign-ins only | project TimeGenerated, UserPrincipalName, Location, IPAddress, AppDisplayName | sort by UserPrincipalName, TimeGenerated asc | serialize | extend PrevTime = prev(TimeGenerated), PrevLocation = prev(Location), PrevUser = prev(UserPrincipalName) | where UserPrincipalName == PrevUser // same user | extend TimeDiffMinutes = datetime_diff('minute', TimeGenerated, PrevTime) | where TimeDiffMinutes < 60 // within 1 hour | where Location != PrevLocation // different country | where PrevLocation !has "VPN" // exclude known VPN | project UserPrincipalName, PrevLocation, Location, TimeDiffMinutes, IPAddress, AppDisplayName | where TimeDiffMinutes > 0
SPL — DGA Detection (DNS NXDOMAIN Storm) T1568.002 · DGA
| Detect DGA bots: high NXDOMAIN ratio + high-entropy domains index=dns sourcetype=dns | eval domain_len = len(query) | where domain_len > 20 /* long labels typical of DGA */ | stats count as total, sum(eval(if(reply_code="NXDOMAIN",1,0))) as nxdomain_count by src_ip, dest | eval nxdomain_ratio = round(nxdomain_count / total * 100, 1) | where nxdomain_ratio > 70 /* >70% NXDOMAIN = DGA bot */ | where total > 50 /* minimum query volume */ | sort -nxdomain_ratio | table src_ip, dest, total, nxdomain_count, nxdomain_ratio
SPL — Ransomware: File Encryption Storm T1486 · Ransomware
| Detect bulk file rename/write activity typical of ransomware encryption index=sysmon EventCode=11 /* Sysmon FileCreate events */ | stats count as file_writes, dc(TargetFilename) as unique_files, values(TargetFilename) as files by Image, ComputerName, _time span=1m | where file_writes > 100 /* >100 writes/minute */ | join Image [ search index=sysmon EventCode=1 | table Image, CommandLine, ParentImage ] | eval has_ransom_ext = if( match(files, "\.(locked|encrypted|enc|cry|xxx|zzz)$"), 1, 0 ) | where has_ransom_ext == 1 OR file_writes > 500 | table ComputerName, Image, CommandLine, file_writes, has_ransom_ext
SPL — Pass-the-Hash Detection T1550.002 · PtH
| Pass-the-Hash: NTLM auth with different source IP than expected | EventID 4624 LogonType 3 (network) + NTLM + no password hash index=wineventlog EventCode=4624 LogonType=3 AuthenticationPackageName=NTLM | stats count as auth_count, dc(IpAddress) as unique_sources, values(IpAddress) as src_ips, values(TargetUserName) as users by ComputerName, WorkstationName, _time span=10m | where unique_sources > 1 /* same account, multiple IPs */ | lookup known_admin_hosts ComputerName OUTPUT is_admin_host | where is_admin_host != "true" | table ComputerName, users, src_ips, auth_count, unique_sources

Hunt 01 — T1053.005
SCHEDULED TASK PERSISTENCE HUNT
Hypothesis: Attacker created a scheduled task pointing to a non-standard executable path (AppData, Temp, ProgramData) to maintain persistence.
KQL // Hunt: Scheduled tasks in suspicious paths SecurityEvent | where EventID == 4698 // task created or EventID == 4702 // task modified | extend TaskXML = extractjson( "$.EventData.TaskContent", EventData ) | where TaskXML contains @"\AppData\" or TaskXML contains @"\Temp\" or TaskXML contains @"\ProgramData\" or TaskXML contains "powershell" or TaskXML contains "mshta" | project TimeGenerated, Computer, SubjectUserName, EventID, TaskXML | order by TimeGenerated desc
POWERSHELL — LIVE # Hunt scheduled tasks on live system Get-ScheduledTask | Where-Object { $_.State -ne 'Disabled' } | ForEach-Object { $task = $_ $action = $task.Actions | Select-Object -First 1 [PSCustomObject]@{ Name = $task.TaskName Path = $task.TaskPath Execute = $action.Execute Arguments= $action.Arguments } } | Where-Object { $_.Execute -match 'AppData|Temp|ProgramData|powershell|mshta|wscript' } | Format-Table -AutoSize
Hunt 02 — T1218 / T1059
LOLBIN ABUSE HUNT
Hypothesis: Attacker is using signed Windows binaries (LOLBins) to execute malicious code, evading application whitelisting.
KQL // Hunt: Living-Off-The-Land Binary (LOLBin) execution with suspicious patterns let lolbins = dynamic([ "mshta.exe", "regsvr32.exe", "certutil.exe", "wmic.exe", "bitsadmin.exe", "msiexec.exe", "cmstp.exe", "installutil.exe", "regasm.exe", "regsvcs.exe", "rundll32.exe", "odbcconf.exe" ]); let suspicious_patterns = dynamic([ "http", "\\\\10.", "\\\\192.", // network paths "AppData", "Temp", "ProgramData", // user-writable paths "decode", "urlcache", "split" // certutil abuse patterns ]); DeviceProcessEvents | where FileName in~ (lolbins) | where ProcessCommandLine has_any (suspicious_patterns) | summarize count() by DeviceName, FileName, ProcessCommandLine, InitiatingProcessFileName | order by count_ desc
Hunt 03 — T1087 / T1069
ACTIVE DIRECTORY RECONNAISSANCE HUNT
Hypothesis: Attacker is performing AD recon to identify targets for lateral movement and privilege escalation.
KQL // Hunt: LDAP queries and AD enum activity IdentityQueryEvents | where QueryType in ( "LdapSearch", "SamrSamEnumerateDomainsInSamServer", "Kerberos TGT" ) | summarize QueryCount = count(), UniqueTargets = dcount(QueryTarget), QueryTypes = make_set(QueryType) by DeviceName, AccountName, bin(TimeGenerated, 1h) | where QueryCount > 50 | where UniqueTargets > 10 | union ( // net.exe enumeration commands DeviceProcessEvents | where FileName in~ ("net.exe", "net1.exe") | where ProcessCommandLine has_any ( "group", "user", "accounts", "localgroup", "domain admins", "enterprise admins" ) | project TimeGenerated, DeviceName, AccountName = InitiatingProcessAccountName, QueryCount = 1, UniqueTargets = 1, QueryTypes = pack_array(ProcessCommandLine) ) | order by QueryCount desc

BASH # Malware often deletes itself after execution # But if a process still has the file open, it's recoverable via /proc # Find processes with deleted files open lsof +L1 | grep deleted # Output example: # evil_proc 1337 root 5r REG 8,1 0 1337 /tmp/.hidden (deleted) # Recover from /proc filesystem # PID 1337, file descriptor 5 cp /proc/1337/fd/5 /evidence/recovered_malware sha256sum /evidence/recovered_malware # Now you have the binary even though it was "deleted" # Recover file content from ext4 via debugfs debugfs /dev/sda1 # debugfs commands: lsdel # list deleted inodes undel <inode> /output/file # recover if still allocated # Recover via strings on raw disk image (last resort) strings /dev/sda1 | grep -A 20 "ELF" foremost -i disk.raw -t exe,elf -o /recovered/ # file carving
// 4.6 — Incident Response //
IR Playbooks
Structured response procedures for the most common incident types. Each playbook defines decision gates, evidence preservation steps, containment actions, and escalation paths.
RANSOMWARE INCIDENT: Do NOT reboot affected systems. Do NOT attempt to decrypt. Preserve evidence before any remediation. Notify legal/counsel before any external communication.
Phase 1 — First 30 Minutes
DETECT & CONTAIN
IMMEDIATE ACTIONS (first 15 min) 1. ISOLATE affected hosts → Network isolation (firewall rule or physical disconnect) → Do NOT power off (memory evidence!) → Keep VPN/RDP for responder access 2. ASSESS blast radius → How many hosts show encrypted files? → Any domain controllers affected? → Any backups accessible from network? 3. STOP the spread → Block C2 domains/IPs at perimeter → Disable affected user accounts → Kill malicious processes if still running: taskkill /PID <pid> /F pkill -f malware_name 4. PRESERVE evidence → Run IR collection script (Module 01) → Take memory dump (winpmem) → Screenshot ransom note + wallpaper → Log everything with timestamps
Phase 2 — First 4 Hours
INVESTIGATE
INVESTIGATION PRIORITIES 1. IDENTIFY PATIENT ZERO → First system with encrypted files (MFT timestamps) → First NXDOMAIN burst (DNS logs) → First unusual process execution (Sysmon Ev1) 2. ESTABLISH TIMELINE → Delivery (phishing email timestamp) → Execution (first malware process) → Lateral movement (logon events 4624 to other hosts) → Encryption start (mass FileCreate events) 3. DETERMINE PERSISTENCE → Check all Run keys on all hosts → Check scheduled tasks → Check services → Check WMI subscriptions 4. COLLECT IOCs → Malware hashes → C2 IPs and domains → Ransom note filename pattern → Encrypted file extension (.locked etc.) → Mutex name (from memory forensics)
Decision Tree
TO PAY OR NOT TO PAY — DECISION FRAMEWORK
ASSESSMENT QUESTIONS (answer before any payment decision): 1. Do we have working backups? (offline, tested, recent) YES → Restore from backup. No payment needed. NO → Continue assessment 2. Is there a free decryptor available? Check: https://www.nomoreransom.org YES → Use it. No payment needed. NO → Continue 3. Was data exfiltrated before encryption? (double extortion) → Check network logs for large outbound transfers → Check threat actor's leak site 4. What is the business impact of non-recovery? → Calculate downtime cost vs ransom amount → Involve legal, executive, cyber insurer LEGAL CONSIDERATIONS: → Paying may violate OFAC sanctions if attacker is sanctioned group → Check: https://home.treasury.gov/policy-issues/financial-sanctions → Consult legal counsel before any payment TECHNICAL NOTE: → Payment does NOT guarantee decryption → Ransomware-as-a-Service operators have poor customer service → Decryptors often corrupt files or fail on large sets
APT / BREACH: Assume the attacker is watching. Do NOT alert them via noisy containment. Build a parallel clean environment for communication. The goal is understand the full scope before eviction.
APT Response — Phase 1
SCOPING & SILENT INVESTIGATION
OBJECTIVE: Understand scope without tipping off attacker 1. ESTABLISH SECURE COMMS → Out-of-band communication (personal phones, Signal) → Do NOT use compromised email/Slack for IR → Create fresh accounts in cloud tenant if needed 2. PASSIVE HUNTING (no action yet) → Query SIEM for IOCs across ALL hosts (not just known ones) → Map all C2 beaconing from DNS/proxy logs → Identify ALL compromised accounts (logon from C2 IP) → Do NOT touch malware yet — attacker may have tripwires 3. IDENTIFY ATTACKER OBJECTIVES → What data did they access? (file access logs, DLP) → Where did they pivot to? (lateral movement map) → Do they have persistence on critical systems? → Have they reached AD / domain controllers? 4. DOCUMENT EVERYTHING → Full network diagram of attacker's path → Timeline from first compromise to discovery → All tools/techniques used (ATT&CK mapping)
APT Response — Phase 2
COORDINATED EVICTION
EVICTION: Do it all at once. Give attacker no time to react. 1. PRE-EVICTION CHECKLIST → All persistence mechanisms identified and documented → New credentials prepared for ALL privileged accounts → New domain admin accounts pre-created (offline) → Legal/law enforcement notified if required → Backup integrity verified (offline, isolated) 2. SIMULTANEOUS EVICTION (T=0) → Block all C2 at perimeter (all identified IPs/domains) → Force password reset ALL domain accounts → Revoke ALL Kerberos tickets: klist purge → Rotate ALL service account credentials → Remove ALL persistence (run keys, tasks, services) → Reinstall compromised hosts from known-good image 3. KRBTGT KEY ROTATION (if DC compromised) → Reset KRBTGT password TWICE (defeats golden tickets) → Interval: 10 hours between resets (Kerberos ticket lifetime) → This invalidates all existing Kerberos tickets 4. POST-EVICTION MONITORING → 30-day elevated monitoring for reinfection attempts → Watch for attacker returning via new vector
Phishing Response
EMAIL INCIDENT PLAYBOOK
PHISHING TRIAGE WORKFLOW 1. COLLECT EVIDENCE → Get original email with full headers → Download attachment to isolated sandbox → Extract URLs from email body → Do NOT click links from primary workstation 2. ANALYZE EMAIL HEADERS Received: headers (trace path) Return-Path: vs From: mismatch → spoofing DKIM-Signature: pass/fail SPF: pass/fail/softfail DMARC: pass/fail/none 3. SANDBOX ATTACHMENT/URL Any.run, Hybrid Analysis, or local Cuckoo → Capture: dropped files, network C2, registry 4. ASSESS USER IMPACT → Did user click? (email gateway click tracking) → Did user enter credentials? (check auth logs) → Did malware execute? (EDR alert, Sysmon events) 5. HUNT FOR OTHER RECIPIENTS PowerShell / Microsoft Graph API: Search-UnifiedAuditLog -RecordType ExchangeItem -Operations "Send" — find all sent copies
Phishing — Remediation
CONTAINMENT & RECOVERY
REMEDIATION STEPS 1. PURGE EMAIL FROM ALL MAILBOXES PowerShell (Exchange Online): Search-Mailbox -SearchQuery "subject:'Urgent Invoice'" ` -DeleteContent -Force # Or: Compliance Center → Content Search → Purge 2. BLOCK SENDER/DOMAIN → Add to Exchange blocked senders → Submit URL to Microsoft/Google Safe Browsing → Block domain at DNS/proxy layer 3. IF USER CLICKED/ENTERED CREDS: → Force password reset immediately → Revoke all active sessions (invalidate tokens) → Enable MFA if not already enabled → Check for mail forwarding rules (attacker persistence) → Audit inbox rules: New-InboxRule backdoor common 4. IF MALWARE EXECUTED: → Escalate to full IR playbook → Isolate the endpoint → Preserve memory image → Follow endpoint IR procedure 5. USER COMMUNICATION → Notify user (without blame) → Reinforce reporting culture → Brief department heads if sensitive data at risk
