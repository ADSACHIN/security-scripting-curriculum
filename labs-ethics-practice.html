Lab Exercise Progression
Eight labs ordered by complexity. Complete them in sequence — each builds on the previous. All require an isolated VM environment. Read the entire lab before executing anything.
Before any lab: Revert your VMs to the clean baseline snapshot. Verify no internet connectivity. Take a pre-lab snapshot. Read the full lab objectives before executing a single command.
01
Static Triage — PE File Analysis
REMnux // 45 minutes
Beginner Static
Objective: Perform a complete static triage on an unknown PE sample. Extract file metadata, compute hashes, analyze PE structure, extract IOC-relevant strings, and produce a written triage report. No execution required.
BASH — REMnux # Download a sample from MalwareBazaar (use any tagged 'AgentTesla' or 'Formbook') # Store in: ~/labs/lab01/sample.exe (password: infected) # Step 1: File identification file sample.exe exiftool sample.exe | grep -E "File Type|PE Type|Machine|Compile|Product" xxd sample.exe | head -4 # Check magic bytes manually # Step 2: Hash everything for algo in md5 sha1 sha256; do ${algo}sum sample.exe | tee -a hashes.txt done ssdeep sample.exe >> hashes.txt # fuzzy hash # Step 3: Entropy analysis python3 -c " import math, sys, struct with open(sys.argv[1], 'rb') as f: data = f.read() freq = {} for b in data: freq[b] = freq.get(b,0)+1 H = -sum(f/len(data)*math.log2(f/len(data)) for f in freq.values()) verdict = 'PACKED/ENCRYPTED' if H > 7.0 else 'normal' print(f'Entropy: {H:.3f}/8.0 → {verdict}') " sample.exe # Step 4: String extraction strings -n 8 sample.exe | tee strings_raw.txt strings -n 8 -e l sample.exe | tee strings_wide.txt # UTF-16 cat strings_raw.txt strings_wide.txt | grep -iE \ "http|\.exe|cmd|powershell|reg|HKEY|socket|crypt|ransom|bitcoin" # Step 5: FLOSS — obfuscated strings floss sample.exe --no-static-strings 2>/dev/null # Step 6: PE analysis python3 analyze_pe.py sample.exe # Use script from Module 03 # Step 7: Write your triage report (template) cat > triage_report.md <<'EOF' # Lab 01 Triage Report **Date:** [date] **Analyst:** [name] **Sample Hash:** [sha256] ## File Identification - Type: PE32/PE64 - Compiler: [from exiftool] - Compile timestamp: [suspicious?] ## Entropy Analysis - Overall entropy: X.X — [packed/normal] - Suspicious sections: [list] ## IOCs Extracted ### Network - URLs: [list] - Domains: [list] - IPs: [list] ## Suspicious Imports - [list dangerous APIs found] ## Recommendation - [submit to VT, proceed to dynamic analysis, etc.] EOF

Intermediate Forensic

// 5.4 — Professional Standards //
Ethics, Law, & Responsibility
The technical knowledge in this curriculum is powerful — and carries corresponding responsibility. This section covers the legal framework, ethical principles, responsible disclosure, and the professional code of conduct every practitioner must uphold.

The Legal Framework

Cybersecurity professionals operate within a dense legal landscape. In the United States, the Computer Fraud and Abuse Act (18 U.S.C. § 1030) makes unauthorized access to computer systems a federal crime — including "testing" a system without explicit written permission. The word "unauthorized" is broad: even accessing a publicly reachable server to run security tests without a signed scope-of-work agreement can constitute a CFAA violation.

The United Kingdom's Computer Misuse Act 1990 is similarly broad, with Section 3A criminalizing even the making or supplying of tools intended for computer misuse — meaning that writing exploit code for unauthorized use can itself be criminal. The EU's NIS2 Directive and member-state implementations add further layers, particularly around notification obligations when testing inadvertently touches real systems.

The rule is simple: written authorization, scope documented, signed before you start. If you don't have a signed engagement letter or permission document, you don't have permission.

Responsible Disclosure

When you discover a vulnerability in a third-party product or system — during legitimate research — the responsible disclosure process exists to protect both the public and the researcher. The standard approach: contact the vendor privately with full technical details, give them a reasonable window to patch (typically 90 days, following Google Project Zero's model), and then disclose publicly once the patch is available or the window has passed.

Bug bounty programs (HackerOne, Bugcrowd) formalize this relationship. Always check a target's security.txt or bug bounty scope before testing — many companies explicitly authorize certain research in exchange for responsible disclosure. Never exploit a vulnerability beyond confirming its existence.

Safe Research Practices

All malware research should be conducted in isolated environments, as described in the lab setup section. Specific obligations for responsible researchers: never distribute functional malware samples outside of controlled platforms; always password-protect archives containing samples; check samples for PII or confidential data before uploading to public sandboxes; understand that VirusTotal and similar services may share submitted samples with their network of partners — this can expose sensitive organizational data if malware was collected from a production environment.

The Defender's Obligation

Security knowledge creates an ethical obligation to use it defensively. The skills in this curriculum — understanding injection techniques, analyzing ransomware, building detection rules — exist to protect systems and the people who depend on them. Any use of this knowledge to cause unauthorized harm is a betrayal of the professional trust that makes security research possible.

"With great power comes great responsibility. Security is a profession built on trust — trust from employers, clients, and the public. Destroying that trust through misuse takes seconds; rebuilding it takes years."
