What Is Threat Intelligence
Threat intelligence is not a feed of IP addresses. It is the analytical process of converting raw data into knowledge that drives better security decisions — faster, with higher confidence, at lower cost.

"Intelligence is information that has been collected, processed, and analyzed to meet the needs of policymakers. Threat intelligence is no different — the consumer is the security team, and the policy decision is how to allocate defensive resources."

— Intelligence Community doctrine, adapted for CTI

// From Collection to Retirement //
The IOC Lifecycle
Indicators of Compromise have a lifespan. Treating them as permanent truths — never retiring stale IOCs — is as dangerous as not having them. This section covers the full lifecycle from collection through expiry.

"An IOC that is never retired is not intelligence — it is noise. Every false positive erodes analyst trust in the entire feed, eventually rendering it ignored."

— CTI Operations Principle
Indicator Half-Life — By Type
IOC DECAY RATES
Different indicator types decay at radically different rates. A hash is stable for months; an IP address may be benign infrastructure within 24 hours.
INDICATOR TYPETYPICAL LIFESPANDECAY CAUSERETIREMENT TRIGGER
File Hash (SHA-256) Months to indefinite Malware recompile changes hash STABLE
C2 Domain Days to weeks Attacker rotates infrastructure post-detection 30+ days no activity
C2 IP Address Hours to days Shared hosting reassigned; attacker rotates 7 days unless confirmed active
URL / URI Path Hours to days Attacker changes staging path easily 72 hours if not confirmed
Email Sender Address Single use Free disposable accounts used once Single-use: retire immediately after campaign
JA3 TLS Fingerprint Weeks to months Attacker changes TLS config or tool version MEDIUM stability
Import Hash (imphash) Months Recompile with same libs → same imphash HIGH stability — family-level indicator
YARA Rule Months to years Attacker becomes aware → modifies payload Retire only when false positive rate rises
Automated Lifecycle Management
IOC SCORING & EXPIRY SYSTEM
A scoring system that increases IOC confidence on re-sighting and decays it over time — automatically retiring stale indicators.
PYTHON from dataclasses import dataclass, field from datetime import datetime, timedelta, timezone from enum import Enum import math class IOCType(Enum): HASH = ("hash", 365) # (name, half_life_days) DOMAIN = ("domain", 30) IP = ("ip", 7) URL = ("url", 3) YARA = ("yara", 180) def half_life(self): return self.value[1] @dataclass class IOC: value: str ioc_type: IOCType first_seen: datetime = field(default_factory=lambda: datetime.now(timezone.utc)) last_seen: datetime = field(default_factory=lambda: datetime.now(timezone.utc)) sightings: int = 1 base_score: float = 50.0 # 0-100 def current_score(self) -> float: """Score decays over time, rises with sightings.""" now = datetime.now(timezone.utc) days_since = (now - self.last_seen).days half_life = self.ioc_type.half_life() # Exponential decay decay = math.exp(-0.693 * days_since / half_life) # Sighting boost (log scale — diminishing returns) sight_boost = min(30, 5 * math.log(self.sightings + 1)) return round((self.base_score + sight_boost) * decay, 1) def is_active(self, threshold=10.0) -> bool: return self.current_score() >= threshold def sight(self): """Record a new sighting — boosts score, resets decay clock.""" self.last_seen = datetime.now(timezone.utc) self.sightings += 1 # Usage: ip_ioc = IOC("185.220.101.34", IOCType.IP, base_score=80) print(ip_ioc.current_score()) # 80.0 (fresh) # After 7 days with no sighting: # score ≈ 40.0 (half decayed — half-life=7) # After 14 days: ≈ 20.0 # After 21 days: ≈ 10.0 → retired

STIX 2.1 & TAXII
STIX is the industry-standard JSON schema for representing threat intelligence objects. TAXII is the transport protocol for sharing them. Together they enable machine-readable intel exchange at scale.
STIX 2.1 Object Model
STIX DOMAIN OBJECTS (SDOs)
STIX 2.1 defines 18 SDO types. These are the building blocks of a structured intel product. Each has a required id, type, created, and modified field.
SDO TYPEREPRESENTSKEY FIELDS
indicatorPattern to detect malicious activity (IOC)pattern (STIX pattern), valid_from, kill_chain_phases
malwareMalware family or instancename, malware_types, capabilities, is_family
threat-actorAdversary group or individualname, aliases, sophistication, goals, resource_level
attack-patternA TTP (maps to ATT&CK technique)name, external_references (ATT&CK ID)
campaignA coordinated set of malicious activitiesname, first_seen, last_seen, objective
toolLegitimate software used maliciously (LOLBins)name, tool_types, kill_chain_phases
vulnerabilityCVE or software weaknessname, external_references (CVE), description
identityOrganization, person, or systemname, identity_class, sectors
course-of-actionDefensive action or mitigationname, description, action_type
intrusion-setClustered activity from same actor (APT group)name, aliases, first_seen, goals

STIX Patterns — The Query Language
STIX PATTERN SYNTAX
STIX patterns are used in Indicator objects to define what the indicator detects. They can be converted to Sigma, YARA, or SIEM query language.
STIX PATTERNS # IPv4 address [ipv4-addr:value = '185.220.101.34'] # Domain name [domain-name:value = 'evil-c2.net'] # File hash [file:hashes.'SHA-256' = 'abc123...'] # Email sender [email-message:sender_ref.value = 'ceo@fake.com'] # URL [url:value = 'http://evil.com/stage2.bin'] # Registry key (Windows persistence) [windows-registry-key:key = 'HKCU\\Software\\Microsoft\\Windows\\CurrentVersion\\Run' AND windows-registry-key:values[*].data LIKE '%AppData%'] # Network traffic (process making C2 connection) [network-traffic:dst_port = 443 AND network-traffic:dst_ref.type = 'ipv4-addr' AND network-traffic:dst_ref.value = '185.220.101.34'] # Compound: file AND network (dropped binary + C2) [file:hashes.'MD5' = 'd41d8cd9...'] AND [domain-name:value = 'evil-c2.net'] WITHIN 300 SECONDS
Python STIX2 library:
PYTHON from stix2 import Indicator, ThreatActor, Relationship, Bundle actor = ThreatActor( name="APT-X", aliases=["GhostPanda"], sophistication="advanced" ) ioc = Indicator( name="GHOSTRAT C2", pattern="[domain-name:value = 'evil.com']", pattern_type="stix", valid_from="2025-01-15T00:00:00Z" ) bundle = Bundle(objects=[actor, ioc]) print(bundle.serialize(pretty=True))

Get objects from a collection GET /api/v2/collections/{id}/objects/ GET /api/v2/collections/{id}/objects/? added_after=2025-01-01T00:00:00Z& type=indicator& limit=100 # Returns: STIX bundle with matching objects # 4. Add objects to a collection POST /api/v2/collections/{id}/objects/ Content-Type: application/taxii+json;version=2.1 {bundle of STIX objects}

// Beyond IOCs //
TTP-Based Intelligence
IOC-based intel is reactive — it detects known threats. TTP-based intel is proactive — it detects adversary behavior patterns that persist across tool changes, infrastructure rotations, and campaign evolutions.

"Indicators of compromise are breadcrumbs left behind. Techniques are the footprints of how the adversary moves. Block the breadcrumbs and they leave new ones. Change the terrain and they must change how they walk."

— Detection Engineering Principle
MITRE ATT&CK as Intelligence Language
CONVERTING INTEL TO ATT&CK — ANALYST WORKFLOW
Every piece of threat intelligence should be mapped to ATT&CK technique IDs. This creates a common vocabulary across vendors, teams, and organizations — enabling automatic correlation.
WORKFLOW ############################################################## # INTEL REPORT → ATT&CK MAPPING PROCESS ############################################################## # Step 1: Read incident/malware report # Example raw intel: "The actor sent spearphishing emails with malicious .docx attachments containing macros. Upon execution, PowerShell downloaded a payload and established persistence via scheduled task." # Step 2: Extract behaviors → map to techniques Behavior: Spearphishing with attachment ATT&CK: T1566.001 Behavior: Malicious macro execution ATT&CK: T1204.002 (User Execution: Malicious File) Behavior: PowerShell download ATT&CK: T1059.001 + T1105 (Ingress Tool Transfer) Behavior: Scheduled task persistence ATT&CK: T1053.005 # Step 3: Build ATT&CK Navigator layer # → Visualize which techniques this actor uses # → Compare against your detection coverage # → Gaps = new detection rules needed # Step 4: Write Sigma rules tagged with ATT&CK IDs # (covered in Module 04)
PYTHON — ATT&CK API from attackcti import attack_client # Query ATT&CK for a specific technique client = attack_client() # Get technique by ID tech = client.get_technique_by_name("Scheduled Task/Job") print(tech[0]["external_references"]) # T1053 # Get all techniques used by APT29 apt29 = client.get_group_by_alias("APT29") techniques = client.get_techniques_used_by_group(apt29[0]) for tech in techniques: tid = tech.get("technique_id", "") name = tech.get("technique", "") print(f"{tid}: {name}") # Build ATT&CK Navigator layer programmatically layer = { "name": "APT29 Coverage", "domain": "enterprise-attack", "techniques": [ {"techniqueID": tid, "score": 100, "color": "#cc0000"} for tid in ["T1053.005","T1059.001","T1566.001"] ] } # Export and load at attack.mitre.org/navigator/
Comparative Analysis
IOC VS TTP DETECTION — A REAL EXAMPLE
APT29 (Cozy Bear) used multiple different malware families and C2 infrastructures across campaigns. IOC-based detection missed re-tooled attacks; TTP-based detection caught them all.
APT29 SolarWinds Campaign (2020): IOC-based detection: - Hash of SUNBURST: caught if hash matched - C2 domain (avsvmcloud.com): caught if in blocklist → Missed: Custom Cobalt Strike loaders (different hash) → Missed: New infrastructure (different domains) → Overall: Most orgs didn't detect until SolarWinds disclosed TTP-based detection: T1195.002 — Supply chain compromise via trusted update T1027.005 — Indicator removal (timestamp manipulation) T1078.002 — Valid domain accounts (no brute force — stealthy) T1207 — DCShadow (novel persistence technique) T1550.002 — Pass-the-ticket with Kerberos Golden Tickets → These behaviors persisted across ALL APT29 campaigns → Behavioral rules detected the attack independently of IOCs Lesson: TTP rules are actor-persistent; IOC rules are tool-specific
Coverage Analysis
ATT&CK COVERAGE GAP ANALYSIS
Map your detection rules to ATT&CK and compare against the techniques your top threat actors use — the gaps are your priority detection backlog.
PYTHON import json # Load: your current Sigma rules (tagged with ATT&CK) # Load: techniques used by your priority threat actors your_coverage = {"T1059.001", "T1053.005", "T1055", "T1003.001", "T1071.001"} actor_techniques = { # from ATT&CK / threat reports "APT29": {"T1195.002", "T1207", "T1550.002", "T1059.001", "T1078.002", "T1027.005"}, "FIN7": {"T1566.001", "T1059.001", "T1056.001", "T1071.001", "T1003.001", "T1486"} } for actor, techniques in actor_techniques.items(): gaps = techniques - your_coverage covered = techniques & your_coverage pct = round(len(covered) / len(techniques) * 100) print(f"\n{actor}: {pct}% covered") print(f" Covered: {covered}") print(f" GAPS: {gaps}") # detection backlog # APT29: 17% covered (1/6 techniques) # GAPS: T1195.002, T1207, T1550.002, T1078.002, T1027.005 # → Priority: write detection rules for these 5

MISP API — AUTOMATED IOC ENRICHMENT PIPELINE
MISP (Malware Information Sharing Platform) is the most widely deployed community threat intel platform. Integrating it with your SIEM creates an automated enrichment and alerting pipeline.
PYTHON — MISP API from pymisp import PyMISP, MISPEvent, MISPAttribute misp = PyMISP( "https://your-misp-instance.org", api_key="YOUR_API_KEY", ssl=True ) # Search for events related to an IOC results = misp.search( value="185.220.101.34", type_attribute="ip-dst", to_ids=True # only confirmed malicious ) for event in results: print(f"Event: {event['Event']['info']}") print(f" Tags: {[t['name'] for t in event['Event']['Tag']]}") print(f" TLP: {event['Event'].get('distribution_name')}") # Create a new event from your own investigation event = MISPEvent() event.info = "GHOSTRAT Campaign — 2025-01" event.add_tag("tlp:amber") event.add_tag("misp-galaxy:threat-actor=\"APT-X\"") # Add IOC attributes event.add_attribute("ip-dst", "185.220.101.34", comment="GHOSTRAT C2", to_ids=True) event.add_attribute("sha256", "abc123...", comment="GHOSTRAT loader", to_ids=True) event.add_attribute("domain", "ghostsvc.evil.com") misp.add_event(event) print("Published to MISP community")
PIPELINE — SIEM ENRICHMENT # Automated pipeline: SIEM alert → MISP lookup → enrich from pymisp import PyMISP import json misp = PyMISP("https://misp.org", "KEY") def enrich_alert(alert: dict) -> dict: """Enrich a SIEM alert with MISP context.""" enriched = alert.copy() iocs_to_check = [] # Extract IOCs from alert if "src_ip" in alert: iocs_to_check.append(("ip-src", alert["src_ip"])) if "domain" in alert: iocs_to_check.append(("domain", alert["domain"])) if "file_hash" in alert: iocs_to_check.append(("sha256", alert["file_hash"])) hits = [] for attr_type, value in iocs_to_check: results = misp.search(value=value, type_attribute=attr_type) if results: hits.append({ "ioc": value, "type": attr_type, "events": [e["Event"]["info"] for e in results], "tags": [t["name"] for e in results for t in e["Event"].get("Tag", [])] }) enriched["misp_hits"] = hits enriched["misp_verdict"] = "MALICIOUS" if hits else "UNKNOWN" return enriched

# Analytic Confidence Language # (Based on IC Directive 203) HIGH CONFIDENCE "We assess with high confidence..." Evidence: multiple independent sources, corroborating technical indicators, government attribution Use when: strong evidentiary basis MODERATE CONFIDENCE "We assess with moderate confidence..." Evidence: single reliable source, technical indicators without full context, vendor reporting Use when: good evidence, some gaps LOW CONFIDENCE "We assess with low confidence..." Evidence: unverified, single source, highly circumstantial Use when: early-stage intelligence, plausible but unconfirmed # NEVER use confidence hedges ambiguously: BAD: "It may be possible that..." BAD: "We think perhaps..." GOOD: "We assess with low confidence that X" GOOD: "The evidence suggests Y (MODERATE)" # Always separate: # FACT: "The malware connects to 185.x.x.x" # JUDGMENT: "We assess this infrastructure # is operated by APT-X (HIGH confidence)"
