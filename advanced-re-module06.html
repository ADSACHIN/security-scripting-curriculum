// Packer Analysis //
Packers, Cryptors & Protectors
Three categories of PE protection with escalating analysis difficulty. Recognizing which type you're facing determines the unpacking strategy.
Type 01
PACKERS
Compress the PE payload. Small stub decompresses at runtime. Entropy 6.5–7.5. Usually a single compression algorithm (LZMA, ZLIB, aPLib). OEP method works cleanly.
# Common packers: UPX → trivial: upx -d sample.exe MPRESS → signature: PE section ".MPRESS1" aPLib → aPack packer (aPlib decompression) NSIS → Nullsoft installer wrapper # Detection (Detect-It-Easy): die sample.exe # Shows: "UPX 3.96 [LZMA, NRV2B]" # Or: "Packer: MPRESS 2.19"
Type 02
CRYPTORS
Encrypt the payload (XOR, RC4, AES, custom). Decryption key may be hardcoded, derived from system info, or fetched from C2. Entropy 7.0–8.0. OEP method works but you need the decrypted memory.
# XOR cryptor (simplest): for i in range(len(encrypted)): plain[i] = encrypted[i] ^ key[i % len(key)] # Finding the key in x64dbg: 1. BP on decryption loop (high RIP activity) 2. Inspect ECX/RCX (key byte) at XOR instruction 3. Run until loop completes 4. Dump decrypted region
Type 03
PROTECTORS
Commercial protection (Themida, VMProtect, Obsidium). Add anti-debug, anti-VM, code virtualization, import obfuscation. OEP method harder — often virtualized. May require specialized tools.
# VMProtect — converts code to VM bytecode: Original x86: PUSH EAX ADD EAX, 4 RET VMProtect equivalent: CALL vm_entry ← enters VM DB 0x3A, 0x1C, 0x08 ← custom bytecodes DB 0x07, 0x2F, 0xB4 ← VM interpreter reads these # Must either: emulate the VM, or # trace execution and reconstruct logic
Packed PE Anatomy
READING THE PE HEADER OF A PACKED SAMPLE

A packed PE has a characteristic structure that distinguishes it from a clean binary. Learn to read these signals immediately:

# pefile analysis of packed sample import pefile pe = pefile.PE("packed.exe") # Signal 1: Few or no imports for e in getattr(pe, "DIRECTORY_ENTRY_IMPORT", []): print(e.dll.decode()) # Legitimate PE: 10+ DLLs # Packed PE: only kernel32.dll + LoadLibraryA # (just enough to bootstrap decompression) # Signal 2: Small .text, large unknown section for s in pe.sections: name = s.Name.decode().strip('\x00') e = s.get_entropy() print(f"{name}: entropy={e:.2f}") # UPX: .text entropy ~7.8 (compressed code) # UPX0/UPX1 sections instead of normal names # Signal 3: Entry point in unusual section ep = pe.OPTIONAL_HEADER.AddressOfEntryPoint for s in pe.sections: if s.VirtualAddress <= ep < s.VirtualAddress + s.Misc_VirtualSize: print(f"EP in section: {s.Name.decode()}") # Packed: EP in ".packed", ".prot", "UPX1" # Clean: EP in ".text"

Anti-Analysis: Detection & Defeat
Malware authors invest heavily in making analysis painful. This section catalogs every significant anti-analysis technique with its exact implementation signature and the specific bypass for each.
Anti-Debug 01
IsDebuggerPresent / CheckRemoteDebuggerPresent
# Reads PEB.BeingDebugged flag (offset 0x2) # Returns 1 if debugger attached # Malware code (conceptual): if IsDebuggerPresent() != 0: ExitProcess(0) # silently die # Assembly equivalent: CALL IsDebuggerPresent TEST EAX, EAX JNZ exit_routine ← jump if debugger BYPASS OPTIONS: 1. ScyllaHide plugin → hooks return value → always 0 2. Manual: find the CALL, patch JNZ → JZ (flip branch) 3. Manual: find PEB.BeingDebugged → write 0 at offset 2 x64dbg: Ctrl+G → fs:[30]+2 → write 0x00 4. Use Scylla anti-debug profile in x64dbg
Anti-Debug 02
NtQueryInformationProcess
# Queries debug port — more stealthy than IsDebuggerPresent # ProcessInformationClass = 7 (ProcessDebugPort) # Returns non-zero if debugger attached # Often used as secondary check after IsDebuggerPresent # because more sophisticated malware assumes analysts # patch the first check BYPASS: 1. ScyllaHide hooks NtQueryInformationProcess → returns 0 for ProcessDebugPort query 2. Manual: BP on NtQueryInformationProcess → after call, inspect return buffer → if ProcessInformationClass==7, patch buffer to 0 3. x64dbg: set conditional BP: COND: [esp+8] == 7 (ProcessDebugPort param) ACTION: EAX = 0 after return
Anti-Debug 03
OutputDebugString / CloseHandle Tricks
# OutputDebugString trick: # Clears last error before call # If debugger attached: GetLastError() returns 0 # If no debugger: GetLastError() returns non-zero SetLastError(0xDEAD) OutputDebugString("test") if GetLastError() == 0: # Debugger is present # CloseHandle trick: # Pass invalid handle to CloseHandle # Without debugger: exception, handled silently # With debugger: EXCEPTION_INVALID_HANDLE raised and debugger may intercept it BYPASS: 1. ScyllaHide patches OutputDebugString behavior 2. CloseHandle: set x64dbg to not break on that exception Options → Exceptions → Add 0xC0000008 (INVALID_HANDLE exception code)
Anti-Debug 04
INT3 / Software Breakpoint Detection
# Malware reads its own code looking for 0xCC (INT3) # Software breakpoints replace bytes with 0xCC # If any 0xCC found → debugger present # Malware implementation: crc32_start = &check_function crc32_end = crc32_start + 100 expected_crc = 0xDEADBEEF # precomputed actual_crc = compute_crc32(crc32_start, 100) if actual_crc != expected_crc: # Breakpoint or patching detected BYPASS: 1. Use HARDWARE breakpoints instead of software (hardware BPs don't write 0xCC — use DR0-DR3) x64dbg: right-click → Set Hardware BP → Execute 2. NOP the CRC check: find the CMP+JNZ → NOP both 3. Timing Attacks — The Subtlest Defense
RDTSC / TIMING-BASED ANTI-DEBUG
RDTSC (Read Time-Stamp Counter) reads the CPU cycle counter. The delta between two RDTSC calls reveals if code ran slowly — as it does under a debugger. This is extremely difficult to defeat without hardware breakpoints.
# RDTSC timing check — assembly: RDTSC ; read TSC → EDX:EAX MOV [saved_tsc], EAX ; save low 32 bits CPUID ; serializing fence ; ... code to time ... RDTSC ; read again SUB EAX, [saved_tsc] ; delta CMP EAX, 0x100000 ; threshold (~1M cycles) JA debugger_detected ; too slow = debugger # With debugger: each breakpoint adds 10M+ cycles # Without debugger: 100-1000 cycles for same code
BYPASS STRATEGIES: 1. ScyllaHide patches RDTSC → returns fake value Small consistent delta regardless of step time 2. x64dbg: set RDTSC hook Options → Engine → Try to fix RDTSC 3. Hardware breakpoints instead of software Hardware BPs don't stall CPU Use x64dbg hardware BP: Ctrl+H 4. NOP the comparison: Find CMP EAX, [threshold] Find subsequent JA/JNE → NOP it Code jumps past the anti-debug exit 5. Patch the threshold: Find the comparison constant (0x100000) Replace with 0xFFFFFFFF Now threshold is impossibly high → never triggered NOTE: QueryPerformanceCounter is similar — same bypasses apply. ScyllaHide handles both.

// Position-Independent Code //
Shellcode Analysis
Shellcode is position-independent code (PIC) with no import table. It must find Windows APIs at runtime by walking the PEB and hashing export names. Once you understand this pattern, all shellcode becomes readable.
Foundation — How Shellcode Finds APIs
THE PEB WALK — EVERY PIECE OF SHELLCODE USES THIS
Shellcode has no import table. It must find kernel32.dll at runtime by walking the Process Environment Block (PEB) → InMemoryOrderModuleList → iterate DLLs → find kernel32 → walk its export table → find functions by name or hash.
; x86 PEB walk — the universal shellcode bootstrap ; FS:[0x30] = PEB on x86 / GS:[0x60] on x64 MOV EAX, FS:[0x30] ; PEB pointer MOV EAX, [EAX + 0x0C] ; PEB.Ldr MOV ESI, [EAX + 0x14] ; InMemoryOrderModuleList.Flink LODSD ; next module (ntdll) XCHG EAX, ESI LODSD ; next module (kernel32) MOV EBX, [EAX + 0x10] ; kernel32 base address ; Now walk kernel32 export table: MOV EDX, [EBX + 0x3C] ; PE header offset MOV EDX, [EBX + EDX + 0x78] ; export directory RVA ADD EDX, EBX ; export directory VA MOV ECX, [EDX + 0x18] ; number of functions MOV EAX, [EDX + 0x20] ; AddressOfNames RVA ADD EAX, EBX ; AddressOfNames VA ; Now loop: hash each name → compare to target hash
# Python: reproduce PEB walk for analysis import pefile def hash_name(name: bytes) -> int: """Ror13AddHash — most common shellcode hash""" h = 0 for b in name: h = ((h >> 13) | (h << 19)) & 0xFFFFFFFF h = (h + b) & 0xFFFFFFFF return h def build_hash_table(dll_path: str): pe = pefile.PE(dll_path) table = {} for exp in pe.DIRECTORY_ENTRY_EXPORT.symbols: if exp.name: h = hash_name(exp.name + b'\x00') table[h] = exp.name.decode() return table # Build table for kernel32.dll kt = build_hash_table("kernel32.dll") # Now: find mystery hash in shellcode, look up: mystery_hash = 0xEC0E4E8E # from shellcode print(kt.get(mystery_hash)) # → "LoadLibraryA" # Run against all common DLLs to resolve all hashes for dll in ["kernel32","ntdll","ws2_32","wininet"]: table = build_hash_table(f"C:/Windows/System32/{dll}.dll") if mystery_hash in table: print(f"{dll}: {table[mystery_hash]}")

Control Flow Flattening
CFF — RECOGNITION & REVERSAL
Control Flow Flattening (CFF) transforms structured code into a giant switch statement. All basic blocks are dispatched through a central dispatcher based on a state variable. The result looks like spaghetti code with no discernible structure — but it's reversible.
// ORIGINAL CODE (readable): if (condition) { block_A(); } else { block_B(); } block_C(); // AFTER CFF (obfuscated): state = 0x1234ABCD; // initial state while (true) { switch (state) { case 0x1234ABCD: // dispatcher check if (cond) state = 0x5678EF01; else state = 0x9ABCDE23; break; case 0x5678EF01: block_A(); state = 0x4567AB89; break; case 0x9ABCDE23: block_B(); state = 0x4567AB89; break; case 0x4567AB89: block_C(); state = EXIT; break; } }
CFF Recognition signals in Ghidra: - Massive function with hundreds of basic blocks - Central 'dispatcher' block with many outgoing edges - Most blocks end with: [state_var] = constant; JMP dispatcher - Condition blocks → update state, jump back to dispatcher Reversal with D-810 (Ghidra plugin): github.com/jpenalbae/d810 Plugins → D-810 → Run all rules Reconstructs original control flow automatically Reversal with miasm (Python): from miasm.analysis.machine import Machine from miasm.analysis.simplifier import IRCFGSimplifierCommon # Symbolic execution follows all paths through dispatcher # Reconstructs true successors for each block Manual approach: 1. Identify the state variable (written before every dispatcher call) 2. Trace: follow state 0x1234ABCD → find which case handles it 3. Build a map: state_value → block_function 4. Re-read code following the map, not the dispatcher
Polymorphic Engines
POLYMORPHIC CODE — SAME LOGIC, DIFFERENT BYTES
Polymorphic malware mutates its own decryption stub on each infection while keeping the encrypted payload identical. The AV signature of the stub never matches twice.
# Polymorphic engine strategy: # 1. Payload is encrypted with a random key each infection # 2. Decryption stub is REGENERATED with equivalent but # different-looking code each time # SAME logic, different bytes (example): # Version 1: MOV EAX, [key] XOR [payload+ECX], AL # Version 2 (equivalent, different bytes): PUSH [key] POP EBX MOV EAX, EBX XOR [payload+ECX], AH ← uses AH instead of AL # Version 3: MOV EDX, [key] AND EDX, 0xFF ← equivalent to low byte XOR [payload+ECX], DL Analysis approach: Run through decryption loop in debugger Dump decrypted payload after loop completes Payload = same every time → analyze payload, not stub YARA: target payload bytes, not stub bytes
Metamorphic Engines
METAMORPHIC CODE — LOGIC REWRITES ITSELF
True metamorphism rewrites the ENTIRE binary — including the engine itself — using code transformations. This is the hardest class to detect with signatures.
# Metamorphic transformations used: 1. Instruction substitution (equivalent instructions): ADD EAX, 1 ←→ INC EAX MOV EBX, 0 ←→ XOR EBX, EBX ←→ SUB EBX, EBX 2. Junk code insertion (never executes or affects state): PUSH EAX / POP EAX (save/restore same register) MOV ECX, ECX (no-op move) XOR EAX, 0x00000000 (XOR with zero = no-op) 3. Register reassignment: Use EDI instead of ESI throughout All references consistently renamed 4. Code transposition (reorder independent blocks) 5. Subroutine reordering (change function order in binary) Analysis implications: Cannot rely on ANY byte-level signature Must use: behavioral detection (what it DOES) YARA: very hard — small invariant byte patterns only ML: statistical features (entropy per section, etc.) Best approach: sandbox execution → behavioral IOCs
